Purpose

Support both OBS macOS Screen Capture source submodes inside Composition Mode. It snapshots the already-established frame of the focused window at its current size and aspect ratio, locks that exact frame as the recording guide, and lets the user manually enlarge and reposition the same window underneath it. Screen Capture is the default and crops the frozen display down to the guide; Window Capture remains selectable and continues to crop the final resized window with its existing Finish behavior unchanged. Four click-through labels beside the guide edges expose the selected source's current Left, Top, Right, and Bottom crops before Finish.

Problem

OBS can crop either a captured window or a captured display precisely, but finding the correct Left, Top, Right, and Bottom values by eye is slow and error-prone. Both sources should resolve to the same locked guide: Window Capture derives crops from the resized window, while Screen Capture derives crops from the selected display edges. The current guide shows no numeric feedback until Finish, so iterative Window resizing requires repeated Finish attempts and containment errors do not reveal the signed deficit at each invalid edge. In Screen Capture the values are geometrically fixed for the session, but moving and enlarging the window remains useful because it changes the content visible beneath the guide. macOS coordinates may also be expressed in points while OBS consumes source pixels.

Solution

Extend the tested Composition Mode state machine with a capture-source selection and live crop preview owned by the active session. ctrl+alt+cmd+C enters with Screen Capture selected; modal-local S selects Screen Capture and W selects Window Capture, both directly and idempotently. Four edge-anchored labels show only L, T, R, and B. Screen values are calculated once from the frozen display; Window values poll the pinned frame at configurable obsCrop.liveRefreshInterval = 0.1 seconds. Negative margins remain visible with their sign in red, and switching source refreshes all four labels immediately without moving or recreating the guide. Return remains the authoritative validation and copy boundary, while Esc cancels. Clipboard, result modal, containment, and Finish arithmetic remain unchanged.

Scope boundary: there is no OBS WebSocket, IPC, UI automation, source discovery, or automatic OBS mutation. Live labels are advisory presentation derived from the same frozen session inputs; only Return validates and writes the source-prefixed clipboard result.

Fixed Product Contracts

IDContractFailure behavior
COMPOSE-MODE-01Composition Mode is a top-level modal entered with configurable default ctrl+alt+cmd+C, parallel to Window Mode at ctrl+alt+cmd+M. Only one top-level mode may be active; entering the other cleanly exits or cancels the current mode. The active guide session has no automatic timeout; only the post-Finish result presentation uses the configured timeout.Mode acquisition or cross-mode teardown failure leaves no half-entered replacement mode and reports concise visible status through Anodyne-owned presentation.
CROP-SOURCE-01Every new Composition session starts in Screen Capture. While the guide is active, unmodified S selects Screen Capture and unmodified W selects Window Capture. Selection is direct, idempotent, generation-guarded, and changes neither the guide nor the target/window/screen snapshots. A real source change refreshes the existing four edge labels and source status immediately.A failed status or label refresh does not commit a half-applied selection. The prior source, status, and label values remain authoritative and recoverable.
CROP-SCREEN-01Screen Capture treats the frozen starting screen fullFrame() as the OBS source rectangle and crops it down to the frozen guide. Its four values are therefore fixed for the session; resizing or moving the pinned window changes only the content underneath the guide.If the guide is not contained by the frozen screen beyond tolerance, copy nothing, keep the session active, identify the invalid edge, and allow W selection or cancellation.
CROP-WINDOW-COMPAT-01Window Capture retains the existing authoritative behavior exactly: at Finish it validates the same pinned target and stale screen/scale invariants, reads the final window frame, requires that frame to contain the guide, and runs the existing pixel-preserving crop calculation. Live preview values are never reused as Finish results.All existing containment, target-destroyed, stale geometry, pasteboard, teardown, retry, and lingering-result behaviors remain unchanged; advisory polling or label-render failures never copy, finish, move, resize, or silently authorize invalid geometry.
CROP-BASELINE-01Entering Composition Mode snapshots the selected window's exact current frame, including its arbitrary width, height, and aspect ratio; it never resizes or repositions that window.No valid focused/remembered window means no session and a concise alert.
CROP-PIN-01The session remains pinned to the same window object and ID even when focus changes.A missing or replaced target cancels the session and copies nothing.
CROP-MATH-01Crop values are integers in OBS order: Left, Top, Right, Bottom. Both submodes use the same selected-source-to-guide pixel conversion, which preserves the guide's rounded pixel width and height exactly.If any guide edge lies outside the selected source rectangle beyond tolerance, keep the session active, identify the invalid edge, and copy nothing; Window Capture remains correctable by moving/resizing, while Screen Capture may switch to W or cancel.
CROP-SCALE-01Automatic scale is the start screen's finite positive currentMode().scale; a positive configured override wins. The chosen value is frozen for the session.Invalid/missing scale or changed screen identity/full frame/scale cancels as stale geometry.
CROP-UI-01The guide is floating, click-through, fixed in absolute screen coordinates, and visible until Finish or Cancel. Its border width is configurable through immutable obsCrop.guideStrokeWidth and defaults to exactly 1 point. Four compact dark labels are centered beside their matching Left, Top, Right, and Bottom edges, prefer the dimmed exterior, flip just inside when screen bounds leave insufficient exterior room, and remain clamped to the guide canvas. They show only the edge letter and integer crop value—never Result or Scale.Canvas setup/refresh/teardown failures retain native handles for retry-safe adapter cleanup; the entire border, masks, and labels remain explicitly click-through.
CROP-LIVE-MATH-01A pure preview calculation returns all four signed OBS-pixel margins from a source rectangle to the frozen guide. It first evaluates each edge against the same point-space tolerance as Finish, then applies the same within-tolerance clamping, frozen scale, rounded source size, rounded guide-relative origin, rounded guide size, and far-edge derivation. A beyond-tolerance point deficit displays as -max(1, ceil(deficitPoints × scale)); a pixel-derived negative margin remains unchanged. Whenever no point or pixel edge is invalid, preview values equal authoritative calculate() output exactly.Every invalid edge carries an explicit invalid flag and a negative integer display value, remains visible with its sign, and is styled red—even when naïve rounding would produce zero. Invalid rectangles or scale return structured failure and never fabricate values.
CROP-LIVE-01Screen Capture calculates its edge labels once from the frozen display and reuses them unchanged. Window Capture reads only the pinned window frame on each generation-guarded tick and refreshes changed labels at immutable configurable obsCrop.liveRefreshInterval, default 0.1 seconds. Switching S/W refreshes immediately; Screen ticks perform no window-frame read and unchanged values perform no canvas write.A transient poll/read/render failure retains the last complete label set, performs no clipboard or session transition, and retries on the next eligible tick. Target destruction continues through the established cancellation path.
CROP-LIVE-LIFE-01The adapter exclusively owns one repeating live-preview timer during the active guide phase. Finish, Cancel, cross-mode entry, target destruction, stop/reload, and startup rollback stop and release it; result linger owns no preview timer or edge labels. Callbacks guard application generation, session generation, selected source, and retained timer identity.Timer stop or canvas teardown failures retain the exact resource for ordered retry, stale callbacks become no-ops, and no live resource survives a successful cleanup.
CROP-CLIPBOARD-01The copied payload is one line prefixed by the selected OBS source mode: Screen Capture | Left: … or Window Capture | Left: …. The established Left, Top, Right, Bottom, Result, and Scale body and field order remain exact after that prefix. Only the closed screen and window values may produce clipboard output.An invalid or missing source rejects Finish before invoking the pasteboard, leaves clipboard contents unchanged, and cancels the untrustworthy session state through the established teardown path.
CROP-OUT-01While the guide is active, a click-through status modal matching Window Mode remains visible with the locked baseline, selected source, and controls. Return removes the guide, copies the selected-source-prefixed one-line crop format, and changes that modal to the selected source plus the same crop values, result dimensions, and scale for obsCrop.resultDuration additional seconds. Esc cancels an active guide or dismisses the lingering result early; early dismissal never clears the clipboard.Pasteboard failure leaves the guide session active and recoverable, with visible failure status in the persistent modal.
CROP-LIFE-01Reload/stop removes the Composition Mode entry hotkey, all modal bindings including W/S, and every guide canvas with no stale callbacks or resources.Cleanup aggregates failures and repeated :stop() retries retained resources, matching existing Anodyne lifecycle policy.

Implementation Notes

Baseline frame and mode boundary

The guide is not an aspect-ratio preset. Composition Mode copies the current window frame verbatim and freezes it in absolute screen coordinates; any positive width and height are valid. Window Mode remains the optional place to establish a preferred size or ratio before Composition Mode begins. Once Composition Mode is active, it never invokes a window move/resize action and does not expire on the Window Mode timer.

Pixel-preserving crop calculation

Both capture sources use the same generic source-to-guide calculation. Window Capture supplies the target's final frame; Screen Capture supplies the frozen screen full frame. Derive the far-edge crops from rounded pixel rectangles so OBS receives internally consistent integers:

sourceRect = selectedSource == "screen" and frozenScreenFullFrame or finalWindow
sourceW = round(sourceRect.w * scale)
sourceH = round(sourceRect.h * scale)
left    = round((guide.x - sourceRect.x) * scale)
top     = round((guide.y - sourceRect.y) * scale)
resultW = round(guide.w * scale)
resultH = round(guide.h * scale)
right   = sourceW - left - resultW
bottom  = sourceH - top - resultH

This guarantees sourceW - left - right == resultW and the equivalent height invariant. Validate point containment before conversion; clamp only within the documented tolerance.

Fixed Screen crops versus dynamic Window crops

Screen Capture is selected by default because OBS's source is the full display. Its source rectangle, guide, and scale are all frozen at entry, so its four edge labels are calculated once and never change during the session. The owner still resizes and repositions the pinned window because those actions change which window content appears underneath the fixed guide. Window Capture remains different by design: W selects it and live ticks read the current frame every configured interval, while Finish independently reads the final frame again. Switching sources immediately swaps the label set but never moves the window, moves/recreates the guide, or starts a new generation.

Signed advisory preview

Preview math validates the source, guide, and frozen scale, checks every edge in point space, and applies the same tiny containment tolerance before pixel conversion. This is essential because a beyond-tolerance subpixel deficit could otherwise round to zero even though Finish correctly rejects it. The representative Left rule below applies symmetrically to all edges; Right/Bottom raw margins remain derived from rounded pixel sizes:

leftDeficitPoints = max(0, sourceRect.x - guide.x)
leftPointInvalid  = leftDeficitPoints > POINT_TOLERANCE
normalizedGuide   = clampOnlyWithinTolerance(guide, sourceRect)

sourceW = round(sourceRect.w * scale)
sourceH = round(sourceRect.h * scale)
rawLeft = round((normalizedGuide.x - sourceRect.x) * scale)
rawTop  = round((normalizedGuide.y - sourceRect.y) * scale)
guideW  = round(normalizedGuide.w * scale)
guideH  = round(normalizedGuide.h * scale)
rawRight  = sourceW - rawLeft - guideW
rawBottom = sourceH - rawTop - guideH

leftInvalid = leftPointInvalid or rawLeft < 0
left = leftPointInvalid
  and -max(1, ceil(leftDeficitPoints * scale))
  or rawLeft

Every invalid edge therefore has an explicit invalid flag and a negative whole-pixel deficit—at least -1—so it is always displayed in red with its sign. When no edge is invalid, all four raw values are golden-equal to Finish. The preview never mutates the clipboard or authorizes Finish; authoritative calculation still performs target, screen, scale, containment, and copy validation from current native state.

Polling and update policy

The adapter owns a single repeating timer only during the active-guide phase. At the default 0.1-second interval, a Window tick reads the pinned frame, asks the controller for signed values, and writes the existing label elements only when values or validity changed. A Screen tick performs no frame read or canvas write because its preview is cached. A real S/W change bypasses the interval and refreshes source status plus all four labels transactionally. Transient failures retain the last complete label set and retry; generation and timer-identity guards make late callbacks harmless.

Window Capture non-regression boundary

The extension may add advisory polling and labels around the shipped Window Capture branch, but it must not use cached preview values in Finish or change that branch's arithmetic, containment tolerance, target/screen/scale staleness checks, failure recovery, crop-value body/order, guide/status teardown semantics, result timeout, clipboard timing, or cleanup order. Existing golden controller/adapter cases remain authoritative; Milestone 10 adds explicit tests where preview state is stale or invalid but Finish independently re-reads and returns the established result or failure.

Guide/mask geometry

The canvas frame is the stored screen's fullFrame(). Convert the absolute guide to canvas-local coordinates once. The dim mask comprises four non-overlapping rectangles around that local guide, followed by the border and four label-pill/text pairs. Top and Bottom labels center horizontally at their edges; Left and Right center vertically. Each prefers the dimmed exterior with a consistent gap, flips just inside the guide when exterior space is insufficient, and is finally clamped to the canvas. Negative global screen origins therefore require no special rendering beyond the absolute-to-local subtraction. Value/style refreshes update the retained elements in place and do not rebuild the canvas.

Scale policy

obsCrop.scaleOverride == 0 selects screen:currentMode().scale; any finite positive override is used verbatim. Native EasyRes mode is expected to report scale 1, but the manual OBS comparison is authoritative. A mismatch leads to a documented override, not OBS IPC.

Non-goals

  • Changing the established window size when a crop session begins.
  • Forcing 16:9 or any other aspect ratio; presets remain an optional Window Mode preparation step.
  • Showing Result dimensions, Scale, clipboard prefixes, or full output strings inside the four edge labels.
  • Reading OBS source dimensions or scene state.
  • Applying OBS filters/transforms, starting recordings, or automating OBS UI.
  • Supporting a guide that spans displays or changes display scale during one session.
  • Guaranteeing compatibility with legacy/deprecated OBS Window Capture; calibration targets the current macOS Screen Capture source in its Screen Capture and Window Capture submodes.
